## ZZやZQは「タイムアウトしない」

Vim/Neovimの`ZZ`と`ZQ`コマンドをご存じでしょうか。あまり知られていないキーですが、`ZZ`は`:wq`（保存して終了）、`ZQ`は`:q!`（保存せず終了）とほぼ同じことができます。私は`:q`を打とうとして`q:`（コマンドラインウィンドウ）が暴発し、よくパニックになるので、そもそも`:`を打たずに済む`ZZ`/`ZQ`を`:q`系より好んで使っています。

この`ZZ`/`ZQ`は普通のキーマップとは異なる性質があります。普通の複数キーのキーマップでは、プレフィックスを押したあと次のキーまでに時間を空けすぎると発火しません。たとえば「`Q`のあとに`S`で全バッファ保存」のような2キーのキーマップを作っても、`Q`を押して一拍置いてから`S`を押すと、`QS`としては成立しないのです。これは`timeoutlen`という設定で管理されています。

ところが`ZZ`/`ZQ`にはこの制約がありません。`Z`を押してしばらく放置してから`Z`を押しても、ちゃんと「保存して終了」が走ります。同じ2キーのコマンドなのに、自作のキーマップは間を空けると失敗し、組み込みの`ZZ`は失敗しない。この差はどこから来るのでしょうか。

この記事では、その差の正体である`timeoutlen`の仕組みを押さえたうえで、`ZZ` / `ZQ`のように「タイムアウトしないプレフィックスキー」を特定のキーだけ自作する方法を扱います。最終的には[which-key.nvim](https://github.com/folke/which-key.nvim)にメニュー表示まで任せる形に着地しますが、そこに至るまでにNeovim側のいくつかの落とし穴を踏むので、それも含めて順を追って説明します。

## キーマップのタイムアウト（`timeoutlen`）

Vimには、複数キーのマッピングを待つための[`timeoutlen`](https://neovim.io/doc/user/options.html#'timeoutlen')という設定があります。既定値は1000（ミリ秒）です。

これが効くのは、あるマッピングが別の長いマッピングの接頭辞になっているときです。たとえば`Q`単体にも何かを割り当て、さらに`QS`にも割り当てたとします。このとき`Q`を押した瞬間、Vimは判断を保留します。「`Q`単体のコマンドを実行すべきか、それとも次に`S`が来て`QS`になるのを待つべきか」が確定しないからです。Vimはこの曖昧さを解消するために、`timeoutlen`の時間だけ次のキーを待ちます。時間内に`S`が来れば`QS`、来なければ`Q`単体、という具合です。

この待ち時間そのものをなくすだけなら[`notimeout`](https://neovim.io/doc/user/options.html#'timeout')（`timeout`を無効化）という選択肢もあります。ただしこれはグローバルな設定で、すべてのマッピングのタイムアウト待ちを無効化してしまいます。「特定のプレフィックスだけ`ZZ`のように振る舞ってほしい」という目的に対しては効きすぎで、ほかのマッピングの挙動まで巻き込んでしまいます。

## 組み込みプレフィックスがタイムアウトしない理由

ここで`ZZ`に戻ります。なぜ組み込みの`ZZ` / `ZQ`は、間を空けても失敗しないのでしょうか。

鍵は、`timeoutlen`が何を対象にしているかです。`timeoutlen`のhelpには「マッピングされたキー列（mapped sequence）が完成するのを待つ時間」と定義されています（`:help 'timeoutlen'`）。タイムアウトの対象はあくまで**マッピング**であり、Vim組み込みのコマンドではありません。そもそも曖昧さによる待ちは、「2つの**マッピング**が同じ文字列で始まるとき」に生じるものと説明されています（`:help map-ambiguous`）。

一方`ZZ` / `ZQ`は、マッピングではなくVim組み込みのノーマルモードコマンドです（`:help ZZ`。試しに`nmap ZZ`を実行しても`No mapping found`になります）。組み込みコマンドは`timeoutlen`の管轄外なので、`Z`を押したあとは時間制限なく次のキーを待ちます。`Z`単体ではコマンドにならず、`ZZ`か`ZQ`の最初の1文字としてのみ機能するため、Vimはコマンドが完成するまで待ち続けます。間を空けても`ZZ`が成立するのは、この待ちにタイムアウトがないからです。

やりたいのは、この「単体では完結せず、次のキーを無期限に待つ」挙動を、自分の好きなキー（ここでは`Q`）で再現することです。

## 手動で`Z`プレフィックスを再現する

素直な発想だと、`QS`や`QW`を`vim.keymap.set`でそれぞれ定義したくなります。しかし前述のとおり、それだと`Q`が複数マッピングの接頭辞になり、`timeoutlen`待ちが発生します。

発想を変えて、接頭辞関係そのものを作らないようにします。具体的には、マッピングするのは`Q` 1個だけにして、`Q`を「押された瞬間に次のキーを自分で読みに行く函数」に割り当てます。

```lua
vim.keymap.set("n", "Q", function()
  --次の1キーを無期限に待つ
  local ok, char = pcall(vim.fn.getcharstr)
  if not ok then
    return -- <C-c>などで中断されたら何もしない
  end
  local key = vim.fn.keytrans(char)

  local actions = {
    S = function()
      vim.cmd("wa")
    end, --全バッファ保存
    W = function()
      vim.cmd("xa")
    end, --保存して全終了
    Z = function()
      vim.cmd("qa!")
    end, --保存せず全終了
  }

  local action = actions[key]
  if action then
    action()
  end
end, { desc = "Q-prefix" })
```

ポイントは2つあります。

- マッピングは`Q` 1個だけなので、接頭辞の曖昧さがなく`timeoutlen`に乗らない。
- 函数の中で[`getcharstr()`](https://neovim.io/doc/user/builtin.html#getcharstr%28%29)が次のキーを無期限に待つ。これが`ZZ`の「`Z`のあとを無期限に待つ」挙動そのものになる。

次のキーの取得に`getchar()`ではなく`getcharstr()`を使い、さらに[`keytrans()`](https://neovim.io/doc/user/builtin.html#keytrans%28%29)を通しているのは、`<Esc>`のような特殊キーを`"<Esc>"`という素直な文字列として扱うためです。`getchar()`が返す生の数値やバイト列を直接扱うより、分岐や表示が安定します。`pcall`で囲んでいるのは、`<C-c>`による中断（`getcharstr()`が例外を投げる）を握りつぶして静かにキャンセルするためです。

:::message
`Q`を上書きすることには副作用があります。Neovimの組み込みの`Q`は、VimのExモードではなく直前に記録したマクロ（レジスタ）を再生するコマンドに変わっています（`:help Q`）。今回のように`Q`を別の用途に使うなら、この組み込みの`Q`は捨てることになります。あとで触れますが、この組み込み`Q`の存在が後半の落とし穴に効いてきます。
:::

## which-keyでメニューを出したい

ここまでで目的の挙動は手に入りました。ただ、コマンドが増えてくると「`Q`を押したら何が選べるのか」を一覧したくなります。これを担うのがwhich-key.nvimです。`Q`を押すとサブコマンドの一覧がポップアップで出てくる、あの仕組みです。

そこで、`Q`の手動ディスパッチはwhich-keyに任せて、`QS`などを普通のキーマップとして定義し、`Q`をグループとして登録する、という素直な書き方を試したくなります。

```lua:lua/config/keymaps.lua
require("which-key").add({ { "Q", group = "Q-Commands" } })

vim.keymap.set("n", "QS", function()
  vim.cmd("wa")
end, { desc = "Write all buffers" })
vim.keymap.set("n", "QW", function()
  vim.cmd("xa")
end, { desc = "Write & quit all buffers" })
vim.keymap.set("n", "QZ", function()
  vim.cmd("qa!")
end, { desc = "Force quit all buffers" })
```

ところが、これだけでは動きません。`Q`を押してもメニューが出ず、それどころかエラーが出ます。

```
E354: Invalid register name: '^@'
```

これは2つの事実が重なって起きています。

ひとつめは、which-keyの`<auto>`（既定のトリガー設定）が、単キーの大文字のプレフィックスを`Z`以外は自動でトリガー登録しないという点です。which-key v3のソース（`lua/which-key/buf.lua`の`is_safe`）には、単キーのプレフィックスについて「小文字は`g`と`z`だけ、大文字は`Z`だけを安全とみなす」というガードがあります。任意の単キーを勝手に乗っ取らないための安全策です。このため`Q`はトリガーにならず、which-keyは`Q`に反応しません。

ふたつめは、さきほどの組み込みの`Q`です。which-keyが`Q`を拾わないので、`Q`押下は素通りして組み込みの「直前のマクロを再生」が発火します。一度もマクロを記録していなければ、再生対象のレジスタが空（`^@`）なので`E354`になる、というわけです。

## `Q`を明示的にトリガー登録する

解決策は、`<auto>`の安全ガードに引っかからない経路で`Q`をトリガー登録することです。which-keyの`is_safe`は、`<auto>`経由では大文字単キーを`Z`以外弾きますが、`opts.triggers`に手で書いた明示トリガーに対してはこのガードを適用しません（同じ`buf.lua`の手動トリガー処理が、ガードなしの判定を通ります）。

つまり、`opts.triggers`に`Q`を明示的に足せばよいということになります。

```lua:lua/plugins/which-key.lua
return {
  "folke/which-key.nvim",
  opts = {
    triggers = {
      { "<leader>", mode = { "n", "v" } },
      { "g", mode = { "n", "v" } },
      { "Q", mode = "n" },
    },
  },
}
```

これで`Q`がwhich-keyのトリガーになり、押すとメニューが出て、`QS`などが選べるようになります。明示トリガーの`Q`キーマップ自体が組み込みの`Q`を上書きするので、マクロ再生の誤発火も同時に解消されます。

なお、ここでは`<auto>`を使わず`<leader>` / `g` / `Q`だけをトリガーに絞っています。`<auto>`を有効にすると、ほかのプレフィックスの体感まで変わってしまうので、必要なものだけ明示する形にしています。

## which-keyは内部で何をしているのか

ここで少し寄り道して、なぜwhich-keyだと「タイムアウトしないプレフィックス」になるのかを見ておきます。これがわかると、手動の`getchar`版とwhich-key版が内部的には同じことをしていると納得できます。

which-key v3は、トリガーとなるキーを`nowait`付きのキーマップとして登録します（`lua/which-key/triggers.lua`）。`nowait`は「より長いマッピングを待たずに即発火せよ」という指示なので、トリガーキーの段階で`timeoutlen`待ちが消えます。そして発火後は、which-key自身が`getcharstr()`を使った独自のループ（`lua/which-key/state.lua`）で次のキーを無期限に待ちます。

つまりwhich-keyは、受け身のポップアップ表示器ではなく、プレフィックスキーの入力処理を自分で乗っ取るプラグインです。やっていることは、さきほど手で書いた「`Q`を即発火する完結マッピングにして、本体で`getcharstr`をブロックする」という技法とまったく同じで、それを汎用化してポップアップを足したものだと言えます。

この視点で並べると、3つの方式が全部同じ原理だとわかります。

| 方式                | 「次キー待ち」を担うもの                              | `timeoutlen`                            |
| ------------------- | ----------------------------------------------------- | --------------------------------------- |
| 組み込み`ZZ` / `ZQ` | Vimのコマンドパーサ（マッピングでないので無期限待ち） | 無関係                                  |
| 手動`getchar`       | 自分で`Q`を函数化し、本体で`getcharstr`をブロック     | 回避（マッピングが`Q` 1個で曖昧さなし） |
| which-keyトリガー   | which-keyの`nowait`トリガー＋独自`getcharstr`ループ   | 回避（上記を一般化＋ポップアップ）      |

共通する核は、「プレフィックスを即確定する完結マッピングにして、その本体で次のキーをブロック取得する。Vimのマッピング・タイムアウト式の曖昧性解消に頼らない」という点です。`ZZ`、手動`getchar`、which-keyのいずれもタイムアウトしないのは、この同じ原理によるものです。

## `expand`はタイムアウトの解ではない

which-keyには[`expand`](https://github.com/folke/which-key.nvim)という、グループの子要素を動的に生成する機能があります。これを使うと、`QS`などを実キーマップとして定義せず、`Q`グループの定義の中に函数として直接書けます。

一見すると「これがタイムアウトを消す仕組みなのか」と思いたくなりますが、そうではありません。`expand`は子要素をどこから供給するか（実キーマップか、動的生成した函数か）を変えるだけで、タイムアウトとは無関係です。タイムアウトしない性質は、前述のとおりwhich-keyのトリガー機構（`nowait`＋独自ループ）が元々持っているもので、`expand`を使うかどうかとは独立しています。

`expand`が消してくれるのは、タイムアウトではなく**手動ディスパッチ**のほうです。とはいえ、今回のようにコマンドが静的に決まっているなら、実キーマップとして定義するほうが単純で、`:nmap Q`などで素直に確認できます。`expand`が活きるのは、子要素を状況に応じて動的に変えたいときです。

## おまけ: command-line windowとの相性

`Q`プレフィックスにコマンドラインウィンドウ（`q:`）を開くサブコマンドを足すと、ひとつ小さな問題が起きます。コマンドラインウィンドウの中でも`Q`がプレフィックスとして反応してしまい、そのウィンドウを閉じる手段として`Q`を素直に使えなくなります。

これは`CmdwinEnter`の自動コマンドで、そのウィンドウの中だけ`Q`を`:q`（ウィンドウを閉じる）に戻すことで解決できます。

```lua
vim.api.nvim_create_autocmd("CmdwinEnter", {
  callback = function()
    vim.api.nvim_buf_set_keymap(0, "n", "Q", ":q<CR>", { noremap = true, nowait = true })
  end,
})
```

バッファローカルなキーマップなので、コマンドラインウィンドウを抜ければ通常の`Q`プレフィックスに戻ります。

## まとめ

`ZZ` / `ZQ`がタイムアウトしないのは、`Z`が単体では完結しないプレフィックスで、`timeoutlen`の対象であるマッピングとは別の仕組みで次のキーを無期限に待っているからでした。同じ挙動を自作のキーで再現するには、プレフィックスを「即発火する完結マッピング」にして、その本体で次のキーをブロック取得すればよく、これは手で`getcharstr`を書いても、which-keyにトリガー登録しても、内部的には同じことをしています。

which-keyに任せる場合の注意点として、`<auto>`は大文字単キーを`Z`以外トリガー化しないため、`Q`のようなキーは`opts.triggers`に明示登録する必要があり、これを怠ると組み込みの`Q`（マクロ再生）が誤発火して`E354`になる、という点を押さえておくと安全です。
