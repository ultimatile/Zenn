## 「機械的」と「文脈依存」という振り分け

AIコーディングエージェント向けのスキル集 [development-skills](https://github.com/ultimatile/development-skills) を作っています。
その中の `done-check` は、コードを対象にした監査スキルです。
エージェントがタスクを「完了した」と宣言する前に、変更差分を品質項目のリストへ照らして点検します。
品質項目の一つひとつには、「機械的」か「文脈依存」のどちらかのタグが付いています。
この振り分けを説明していたら、それはトヨタ生産方式（Toyota Production System, TPS）の「自働化」じゃないか、と言われました。

二つのタグは、こう分かれています。

- 機械的は、判定が決定論的に下る項目。リンターやコマンドの実行結果、字面の照合で白黒がつく。コマンドを実行すれば判定できるなら、それは機械的だ。判断する主体すら要らない。
- 文脈依存は、意図・計画・レビュー履歴といった文脈を理解しないと判定できない項目。決定論では割り切れず、理解する主体が判断する。

プログラミングでは、構文（syntax）と意味（semantics）が区別されます。
前者は抽象構文木や正規表現で扱え、機械的に判定できます。
後者はそれらでは判定できず、基本的にプログラミング技術スタックの外側として扱われます。
機械的と文脈依存の振り分けも、おおよそこの線に沿っています。

次のコードの欠陥は機械的です。

```python
def total(items):
    return sum(item.price for item in itmes)  # itmes は items の打ち間違い
```

`itmes` は未定義なので、実行すれば `NameError` になり、型チェッカーやリンターも静かには通しません。
意図を知らなくても、コマンド一つで白黒がつきます。

一方、次のコードの欠陥は文脈依存です。

```python
def total(items):
    return sum(item.price for item in items[:-1])  # 末尾の1件を落としている
```

構文も型も通り、リンターも何も言いません。
ですが `items[:-1]` は最後の1件を黙って除いています。
欠陥かどうかは、total が全件の合計だという想定を知って初めて分かります。

機械的な項目は、エージェントを呼ぶまでもありません。
リンターやコマンドに任せれば、同じ入力にいつも同じ判定が返ります。
やっかいなのは文脈依存のほうで、ここは誰かが理解して判断するしかありません。

その判断を書き手本人にやらせると、曇ります。
自分の書いたコードについて「こう書いたつもり」という思い込みを持っているからです。
そこで判定は、フレッシュコンテキストのサブエージェントに渡します。
サブエージェントは文脈を理解します。
あえて新規の文脈を渡すのは、書き手の思い込みに汚染されない正しい文脈で理解してほしいからです。

この振り分けは、品質監査に限らず自動化一般で現れます。
決定論的に機械が判定できる部分と、理解が要る部分をどこで分けるか、という普遍的な問題です。
以下、その具体例として、自動車の生産ラインで使われてきた自働化を取り上げます。

## 自働化

[自働化](https://ja.wikipedia.org/wiki/自働化)（じどうか）は、[トヨタ生産方式](https://ja.wikipedia.org/wiki/トヨタ生産方式)（TPS）を支える二本柱の一つです。
普通の「自動化」と読みは同じですが、わざと人偏のついた「働」の字を当てています。

語源は、豊田佐吉が発明した自動織機です。
縦糸が切れると機械が自分で異常を検知して止まる仕組みを持っていました。
糸が切れたまま織り続けると、大量の不良の布ができてしまいます。
止まれば不良は1枚で済みます。

肝は、全部を機械にやらせる「自動化」ではない点です。
機械がやるのは異常の検知と自動停止までです。
止まった後に原因を突き止めて直すのは、人間の仕事として残します。
人偏は、その「人が判断に専念する」部分を指しています。

機械的と文脈依存の振り分けは、この自働化の境界そのものです。
機械が決定論的に検知して自動で止まる、これが機械的です。
止まった後に人が文脈を理解して原因を判断する、これが文脈依存です。
前者は機械任せで人もエージェントも要らず、後者では理解する主体が要ります。
織機が糸切れで自動停止し、原因究明を人へ渡すのと、同じ境界が引かれています。

## TPSの用語との対応

自働化の周辺には、同じ「異常で止めて下流に流さない」思想を担う用語が並んでいます。
ハーネスの語彙と並べると、対応がはっきりします。

| TPSの用語 | TPSでの意味 | ハーネス／本スキルでの対応 |
| --- | --- | --- |
| 自働化 | 異常を機械が検知し自動で止める。人は判断と改善に専念する | 機械的／文脈依存のレーン分割そのもの |
| 自動停止 | 全自動ではなく「異常があれば止まる」 | 完了宣言やコミットを無条件に通さず、検知時に止める |
| アンドン | 異常を可視化してラインを止める仕掛け | リンターやフックの失敗、CI の赤、done-check の警告 |
| 後工程に不良を流さない | 各工程で品質を作り込む | 完了宣言の前に止め、腐敗を下流（レビュー・本番）へ流さない |
| ポカヨケ | 源流でミスを起こさせない仕掛け | 決定論的ガードレール（リンター・Hooks） |
| 5なぜ／改善 | 人手による根本原因の追究 | 文脈依存なレーン、gate-miss-to-issue |

TPSの用語はあまり馴染みがないかもしれないので、なじみの薄いものを順に見ていきます。

- アンドンは、生産ラインの異常を電光掲示板やランプで「見える化」し、必要ならラインを止める仕掛け。作業者は異常を見つけると綱（アンドンコード）を引いてラインを止め、応援を呼ぶ。止める権限を現場に与えるのが本質。
- 後工程に不良を流さないは、各工程が自分のところで品質を作り込み、不良を次の工程へ渡さないという原則。止めるのが早いほど手戻りは小さい。
- ポカヨケは、そもそもミスや不良が起きないようにする源流対策。治具の形状で部品の取り付け間違いを物理的に不可能にする、といった工夫を指す。
- 5なぜは、異常の原因を「なぜ？」と5回繰り返して根本原因まで掘る、人手の分析手法。改善（カイゼン）はその結果をラインに反映する活動。

なお、TPSのもう一本の柱はジャストインタイムですが、こちらは在庫と流れの設計の話で、いま見ている「止めて流さない」軸とは別なので立ち入りません。

## TPS・リーン生産方式・リーンソフトウェア開発の系譜

ここで、TPSと聞いてリーンを思い出した人もいるはずです。
ただ「リーン」と一口に言っても、指す層が違います。
[リーン生産方式](https://ja.wikipedia.org/wiki/リーン生産方式)（LPS）は、TPSを欧米の研究者が一般化したものです。
"lean" の語は John Krafcik が1988年に用い、Womack と Jones らの『The Machine That Changed the World』(1990) と『Lean Thinking』(1996) で広く知られるようになりました。
これをソフトウェア開発へ翻案したのが [リーンソフトウェア開発](https://ja.wikipedia.org/wiki/リーンソフトウェア開発)（Lean Software Development, LSD）で、Mary と Tom Poppendieck の同名書 (2003) が起点です。
ソフトウェアの文脈で「リーン」というと、多くはこの LSD や Lean Startup を指します。

TPS→LPS、LPS→LSD という二つのつながりは、それぞれ有名です。
ですが、TPSからソフトウェアまで一続きに辿る人は、案外多くないかもしれません。
そして通して見ると、この記事の主役である自働化が、途中で姿を変えていると分かります。

## 系譜の中の自働化

自働化は、LPSまでは概念として引き継がれます。
ところが LSD の7原則（無駄をなくす／学習を増幅／決定を遅らせる／速く届ける／チームに権限を／統合性を作り込む／全体を最適化する）には、自働化もアンドンも名前として出てきません。
紛らわしいのが「統合性を作り込む（Build Integrity In）」で、これは自働化の子孫ではなく、設計の一貫性（リファクタリングやテストで保つ概念的・知覚的な整合性）を指す別物です。

ただし、自働化が消えたわけではありません。
Poppendieck の続編 (2006) を読むと、「欠陥が出たらラインを止めて原因を直す」という考えは残っていて、そのソフトウェア版のアンドンが TDD や CI のテストだと書かれています（2006年版はこの原則を「品質を作り込む（Build Quality In）」と改名しています）。
つまり自働化は、柱としての地位と名前を失い、止める機構だけが別の原則へ畳み込まれて生き残ったのです。
そして最も痩せたのが、人間の側でした。
TPSで自働化と不可分だった「止める権限を現場に与える文化」は、ソフトウェア版では「チームに権限を」という、より薄いチーム自律へと後退します。

## 二種類の組織論

ここで注意したいのは、TPSもLPSも、組織論に大きなウェイトを置いている点です。
LPSが人間を消したわけではありません。
TPSと同じく、現場の人と組織を中心に置きます。
むしろ、ソフトウェアやエージェントの文脈で薄れるのが、その組織論のほうです。

ただし「組織論」はひとつの塊ではありません。
一方に、機械的な部分があります。
たとえば決裁の権限で、部長は決裁できて課長はできない、と決めてしまえば、あとはそのルールが厳粛に適用されます。
これは決定論的なルールで、最初に見た「機械的」と同じ性格です。
もう一方に、ファジーで感情的な部分があります。
なぜ人はサボるのか、根回し、面子、止めたら責められるのではという心理。
ラインを止めるのが摩擦のある行為になるのは、この感情の側のせいで、TPSが力を注いだのもここでした。

対象がずっと人間だったので、この二つを区別する意味はあまりありませんでした。
人間はどちらも併せ持つからです。
ところが対象がエージェントになると、分離が効いてきます。
権限のような機械的なルールは、コミットゲートや権限設定としてそのまま移ります。
一方で、サボる動機も面子も根回しも、エージェントにはありません。
止めることに心理的な摩擦が無いので、設計でそう決めれば、ただ止まります。

これは、さきほどの自働化と同じ線です。
機械的な半分（検知して止める、権限のルール）は移り、感情的な半分（止める摩擦、人を動かす組織論）は要らなくなります。
製造業の生産方式が最も力を注いだのは後者でしたが、エージェントコーディングではそこが外側に追いやられます。

## まとめ

「機械的」と「文脈依存」という振り分けは、決定論で機械が判定できることと、理解して判断するしかないことの境界という、自動化一般の普遍的な問いに触れています。
トヨタ生産方式の自働化は、その境界を製造業で先に引いた例でした。
TPSからLPS、そしてLSDへと辿ると、自働化の機械的な骨格（検知して止める）は名を変えながら生き残り、人間と組織の側は層を下るほど痩せていきます。
エージェントコーディングはその延長で、止める機構と権限のルールは引き継ぎ、止める摩擦をめぐる組織論はほとんど引き継ぎません。
