## 問題意識

[テストオラクル問題の記事](./test-oracle-problem-numerics)で、テストオラクル問題の大部分はフレーミングの問題であり、機能要件レベルの不変量は仕様から書けると論じた。しかし中間レベル——コンポーネントを合成した結果、不変条件が非自明になるケース——では2つの問題が残っている:

1. **不変性の成立条件がわからない**: SVDの再構成精度 `‖A - UΣV*‖ < ε` は不変性として書けるが、「条件数がいくつ以下のとき成立するか」は別問題
2. **不変性の発見自体が非自明**: SVDは代数的構造がきれいなので不変性が自明に列挙できるが、一般の数値関数ではそうではない

[不良設定の記事](./ill-conditioning-harness-engineering)では、ill-conditioningの判定自体がill-definedであり、ドメイン知識として人間が入れ知恵するしかないと書いた。本記事はこの「入れ知恵」を（部分的に）自動化する枠組みを提案する。

## 3層構造

テストオラクルの自動構築は3層に分かれる:

| 層 | 内容 | 例 | 難易度 |
|---|---|---|---|
| 第1層 | 不変性そのもの | `‖A - UΣV*‖ < ε` | 比較的発見しやすい |
| 第2層 | 不変性の成立条件 | `cond(A) < threshold` | **本記事の対象** |
| 第3層 | 成立条件の数学的導出 | 条件数の定義と理論 | 数学の仕事 |

第1層は不変性ベースのテスト設計で扱った。第3層は数値解析の研究そのもの。第2層は「理論的導出はできなくても、経験的に成立条件を発見する」ことが目標であり、自動化の余地がある。

## 核心的なアイデア: 入力ミューテーションの連続化

ミューテーションテストとの対応で考える:

| | ミューテーションテスト | 微分テストフレームワーク |
|---|---|---|
| 変異対象 | ソースコード（離散的） | 入力値（連続的） |
| 変異操作 | 演算子置換、定数変更等 | 摂動、スケーリング、ULP近傍 |
| 観測対象 | テストのpass/fail | 出力の変化量 |
| 測っているもの | テストスイートの検出力 | 関数の感度構造 |

浮動小数点の「変異」は連続的であり、変異の強度にスペクトルがある。1ULP動かす微小摂動から10倍スケーリングまで、応答が線形か非線形か、閾値的な遷移があるかで、関数の感度構造が見える。

この「感度マップ」が第2層の成立条件を示唆する。

## パイプライン

```
Step 1: 感度データの収集
  ├─ 自動微分（AD）でヤコビアン・条件数的な量を計算
  ├─ 摂動プロトコル（スケーリング、加法的ノイズ、ULP近傍）で応答を観測
  └─ 差分テスト（参照実装があれば）で乖離を収集

Step 2: 成立条件の仮説生成
  ├─ concolic execution: プログラムの分岐構造から離散的な条件を直接抽出
  ├─ LLM: 感度データを見て「何が支配的か」を仮説として生成
  └─ シンボリック回帰: 成立/不成立の境界を簡潔な数式で記述

Step 3: 仮説の反証
  ├─ PBT: 成立条件の外側で不変性が本当に破れるかを確認
  ├─ concolic: 分岐の反対側に到達する入力を構成
  └─ SMT: 形式的に反証 or 証明（可能な範囲で）
```

### 何がジェネリックで何がドメイン依存か

- **ジェネリック**: Step 1の摂動プロトコル、Step 3の反証手法
- **ドメイン依存**: Step 2のLLMへの入力（「条件数を計算してみては？」等の示唆はLLMのドメイン知識に依存）
- **半自動**: ADによる感度計算は汎用だが、「どの量が支配的か」の解釈にはドメイン知識が要る。ここをLLMが埋める

### CFL条件の「再発見」を例に

陽解法の安定性テストを微分テストフレームワークで扱う場合:

1. **ADで感度計算**: `Δt`, `Δx` に対する解のノルムの感度を自動計算
2. **摂動プロトコル**: `Δt`, `Δx` を独立に変化させて発散/非発散を観測（ここで「何を振るか」は人間が指定しない——全パラメータを振る）
3. **成立/不成立の境界**: 発散=NaN/Inf/ノルム爆発は自動検出可能
4. **LLMに感度データを見せる**: 「Δt/Δx²が支配的では？」という仮説が出てくる（かもしれない）
5. **仮説の検証**: 実際に `Δt/Δx²` を固定して他のパラメータを振り、境界が一定かを確認

重要なのは、ステップ4で「Δtとdxを振れ」と人間が指示するのではなく、ステップ2で全パラメータを振った結果をLLMが解釈する点。CFL条件を知っている必要はない。

## メタモルフィックテストとの関係

広義にはメタモルフィックテストの拡張と位置づけられる:

- **メタモルフィックテスト**: 入力変換に対する出力の関係を**検証する**（関係は事前に知っている）
- **微分テストフレームワーク**: 入力変換に対する出力の変化を**観測し、関係と成立条件を発見する**

メタモルフィックテストはMR（メタモルフィック関係）が既知であることを前提とする。微分テストフレームワークはMRの発見と、MRの成立条件の発見を含む。テスト手法というより解析手法であり、テストは最終出力の一つ。

## LLMの役割

LLMは2箇所で登場する:

1. **第1層（不変性の候補発明）**: docstring・型・トレースから不変性候補を列挙
2. **第2層（成立条件の仮説生成）**: 感度データ・差分データから「何が支配的か」を仮説として生成

どちらでも**仮説生成器であって判定器ではない**。真偽判定は検証器（PBT、SMT、concolic等）に渡す。

## concolic executionとの接続

[concolic executionの記事](https://zenn.dev/termoshtt/articles/concolic-execution)で論じられているように、concolic executionはプログラムの分岐構造から条件を直接抽出できる。

微分テストフレームワークにおける役割:

- **ADが捉えるもの**: 連続的な感度（勾配、ヤコビアン）
- **concolicが捉えるもの**: 離散的な分岐条件（`if pivot == 0`、`if iter > max_iter`等）

数値計算コードにはこの両方が混在している。LAPACKのゼロピボット判定は離散的分岐だが、その背後にある条件数の劣化は連続的。両方を組み合わせることで感度構造の全体像が見える。

## 最大のボトルネック: 浮動小数点

パイプラインの各段階で浮動小数点が問題になる:

- **AD**: 浮動小数点演算のADは可能だが、丸め誤差の伝播自体がADの対象にならない
- **concolic**: シンボリック追跡が近似になる。`f(x) == 1` のような変換を伴う比較をソルバーで解くのは困難
- **SMT**: QF_FP理論は遅い

整数算術の世界では第1層〜第3層が既存ツールでそこそこ繋がるが、浮動小数点が入った瞬間にギャップが生じる。

## 既存研究との関係

| 研究方向 | 関係 |
|---|---|
| Verificarlo（自動誤差解析） | Step 1の感度データ収集に近い。ただしテスト設計への接続がない |
| NumFuzz（浮動小数点ファジング） | Step 3の反証に相当。感度構造の理解は目標にしていない |
| MR mining（MR自動発見） | Step 2のMR発見に対応。ただし成立条件まで踏み込んでいるものは少ない |
| LaM4Inv（LLM+BMC） | アーキテクチャ（候補生成→形式検証）は共通。ただし対象はHoare論理のループ不変量であり、テストオラクルとしての不変性とは別の階層 |
| sensitivity-guided testing | 感度情報をテストに使う発想は共通。数値計算の条件数まで踏み込んだ定式化は見当たらない |

**本記事の新規性（仮説）**: 「数値計算テストにおけるoracle problemを、AD駆動の感度解析で第2層（成立条件）まで自動化する」というフレーミング。個別のピース（AD、PBT、concolic、LLM）は既存だが、数値計算のテスト設計を感度構造の観測問題として定式化し、これらを統合するフレームワークの提案は新しい可能性がある。
