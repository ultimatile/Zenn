## はじめに

[Claude Code / Codex ユーザーのための誰でもわかるHarness Engineeringベストプラクティス](https://nyosegawa.github.io/posts/harness-engineering-best-practices-2026/)は、AIコーディングエージェント（Claude Code、Codex等）を安定稼働させるための包括的なガイドです。リンター・Hooks・テストによる決定論的ガードレール、CLAUDE.mdのポインタ設計、E2Eテスト戦略など、実践的な知見が詰まっています。

しかし、テンソルネットワーク計算ライブラリ[Ariadnetor](https://github.com/ultimatile/ariadnetor)の開発を通じて、記事の枠組みにはいくつかの構造的な盲点があることに気づきました。本稿では、記事の議論を補完する形で、ドキュメント管理とナレッジマネジメントの観点から考察します。

## 記事の要約

記事の主張を簡潔にまとめると以下の通りです。

- ハーネスエンジニアリング（CLAUDE.md/AGENTS.mdの継続的改善 + エージェントの自己検証ツール群）がモデル選択より重要
- リポジトリには実行可能なアーティファクト（コード・テスト・リンター設定）とADR（Architecture Decision Records）を置く
- 「現在のシステム構成」「アーキテクチャ概要」といった説明文書は避けるべき
- 仕様や期待動作、制約は可能な限りテストとして表現する
- ADRはステータス（Accepted/Superseded/Deprecated）で有効性を管理する
- CLAUDE.mdは50行以下のポインタとして設計せよ
- 決定論的ツール（リンター・Hooks・テスト）でフィードバックループを閉じよ

これらは概ね正しいのですが、特にドキュメント管理に関する以下の部分に議論の余地があります。

記事はリポジトリ衛生について、置くべきものと置くべきでないものを次のように整理しています（筆者による要約）。

> **置くべきもの**: 実行可能なアーティファクト（コード、テスト、リンター設定）とADR
> **置くべきでないもの**: 「現在のシステム構成」や「アーキテクチャ概要」といった説明文書
> **仕様の扱い**: 可能な限りテストとして表現する

## 仕様書の居場所がない

ADRは「Xを採用した。理由はY」という**決定**を記録するものです。しかしこれはXの定義が読み手と共有されていることを前提としています。

「クリーンアーキテクチャを採用する」というADRは、クリーンアーキテクチャが広く知られた概念だから成立します。ではプロジェクト固有の仕様——例えば独自に定義したインターフェース仕様——を採用する場合はどうでしょうか。ADRに「X仕様に準拠する」と書いても、X仕様の定義がどこかになければ実装できません。

記事のドキュメント分類にはこの層が欠けています。

| 種類 | 記事の扱い |
|------|-----------|
| コード・テスト | 残す（実行可能） |
| ADR | 残す（決定記録） |
| 説明文書 | 避ける（腐敗する） |
| **仕様書** | **テスト化を推奨** |

記事は仕様の居場所として「テスト」を提案しています。テストは実行すれば嘘をつけないので、腐敗耐性が高いという論理です。これは一定の正当性があります。しかし、テストは仕様の**一部**を検証できますが、仕様**そのもの**の代替にはなりません。テストは「何をチェックしたか」は示しますが「なぜそのチェックが必要か」「全体の設計意図」は示せないからです。

仕様書は説明文書とは本質的に異なります。説明文書は「現在のシステムはこう動く」であり、コードが変わると腐敗します。仕様書は「こう動く**べき**」であり、コードが逸脱したら**コード側が間違い**です。仕様書は実装の従属物ではなく、実装が準拠する規範です。

## 「仕様は実装に先んじる」という神話

では仕様書をきちんと管理すれば解決するかというと、そう単純ではありません。現実の開発では仕様と実装は共進化します。

1. 「こういう感じでいけるかも」と実装を始める
2. 実装してみて初めて制約や可能性が見える
3. そこから仕様が結晶化する
4. 結晶化した仕様が次の実装を制約する
5. 新しい実装がまた仕様を揺さぶる

ここで重要になるのがFeasibility Study（FS）です。FSとは「そもそもこれは実現可能か？」を実際に手を動かして検証するプロセスです。仕様の前に実装がある——ただしそれは本実装ではなくFSとしての実装です。

知識の成熟度で整理すると、ドキュメントの適切な器が見えてきます。

| 成熟度 | 実態 | 適切な器 |
|--------|------|----------|
| 仮説 | 「この方法でいけそう」 | Issue / PR上の議論 |
| 検証済み | 「動いた。こう動く」 | コード + テスト |
| 形式化 | 「こう動くべき」 | 仕様文書 |

記事が「置くべきでないもの」として挙げている説明文書の中に、デスクリサーチ（外部ライブラリの調査）、FS記録（検証結果）、仕様書（規範）が未分化のまま混ざっています。これらはそれぞれ腐敗特性が異なり、一括して扱うべきではありません。

## ドキュメント腐敗の実態

理論の話だけでは説得力がないので、実際のプロジェクトでの腐敗を見てみましょう。Ariadnetorのdev-docs/（89ファイル）に対してリンク検証スクリプトを走らせた結果です。

```
$ cargo make doc-lint
Found 31 broken link(s):

  core/contract/autocontract_design.md:76 → raw_fat_tensor_responsibility.md
  core/tensor/tensor_architecture.md:113 → tensor_storage_design.md
  core/lazy/lazy_evaluation_architecture.md:6 → DESIGN_DECISIONS.md
  ...
```

31件のリンク切れ。原因は主に3つです。

1. **アーキテクチャ変更後のリンク未修正** — MLIR/IREEを除去してPure Rustに移行した際、archiveディレクトリに移動したファイルへの参照が残存
2. **リネーム後の未修正** — `tensor_storage_design.md`→v2へのリネーム、型名の変更
3. **存在しないファイルへの参照** — `element_type_design.md`は3ファイルから参照されているがどこにも存在しない

さらに深刻なのは、コードとの乖離です。

- ドキュメントに`contract_naive`メソッドの記述があるが、コード上には存在しない
- `DenseTensor::gemm_f64`がメソッドとして記述されているが、実際はスタンドアロン関数に変更済み
- `permute_hppt`（typo）が文書にあるが、実装は`permute_naive`のみ

手動でのパージ（一度MLIRパージを実施してarchiveに押し込んだ経緯がある）はスナップショットに過ぎず、コードが変わるたびに同じ問題が再発します。

## 機械的な腐敗検出

記事の「決定論的ツール」の発想を借りれば、ドキュメントの腐敗も機械的に検出すべきです。最も費用対効果が高いのはリンク検証です。

```python
#!/usr/bin/env python3
"""Check dev-docs markdown files for broken internal links."""

import re
import sys
from pathlib import Path

LINK_RE = re.compile(r"\[([^\]]*)\]\(([^)]+)\)")

def check_links(docs_dir: Path):
    errors = []
    for md_file in sorted(docs_dir.rglob("*.md")):
        for line_num, line in enumerate(
            md_file.read_text().splitlines(), 1
        ):
            for match in LINK_RE.finditer(line):
                link = match.group(2)
                if link.startswith(("http://", "https://")):
                    continue
                target = link.split("#")[0]
                if not target:
                    continue
                resolved = (md_file.parent / target).resolve()
                if not resolved.exists():
                    rel = md_file.relative_to(docs_dir)
                    errors.append((str(rel), line_num, link))
    return errors
```

完璧に腐敗を防ぐのは無理ですが、「壊れたリンク」という最も明確な腐敗シグナルを自動検出するだけでもかなり違います。記事が提唱するフィードバック速度の階層——PostToolUse（ミリ秒）> プリコミット（秒）> CI（分）> 人間レビュー（時間）——のCI層に、ドキュメントのリンク検証を組み込むのは自然な拡張です。

## ADR/Spec/FSの渾然一体問題

腐敗検出は対症療法です。根本的な問題は、1つのドキュメントの中にADR・Spec・FSが混在していることです。

例えば`scalar_trait_design.md`には以下が混ざっています。

- **FS部分**: Rustコンパイラのエラー（E0592）がなぜ起きるかの調査
- **ADR部分**: 補助トレイトパターンを選択した理由、他の選択肢（ニュータイプ、拡張トレイト）との比較
- **Spec部分**: `Scalar` traitの定義、APIシグネチャ

これらは腐敗特性が異なります。

| 部分 | 腐敗するか | 理由 |
|------|-----------|------|
| FS | しにくい | 「調査した結果こうだった」は事実 |
| ADR | しにくい | 「当時こう判断した」は事実 |
| Spec | **する** | コードが変わると乖離する |

1ファイルに混在していると、どこが腐っていてどこが安全か判別できません。

## 仕様のスキル化——理想と共進化の壁

安定した仕様はエージェントの「スキル」として外部化するのが理想的です。Claude Codeのスキル機能を使えば、仕様の全文を必要な時だけコンテキストにロードできます。CLAUDE.mdのポインタ方式よりも精密で、エージェントが仕様を参照するたびに正確な情報が注入されます。

実際にAriadnetorでは、テンソル演算のインターフェース仕様（TCI Spec）をスキル化しています。これは形式化が完了した安定した仕様だから成立しました。

しかし、スキル化には前提条件があります。仕様が安定していなければならないという点です。先述の通り、仕様と実装は共進化します。実装中のコンポーネントの仕様をスキル化しても、実装のたびにスキルの中身を更新する必要が生じ、メンテナンスコストがスキル化の利点を上回ります。

つまりスキル化は**仕様のライフサイクルの最終段階**で行うべきものです。

```
FS → 実装用Spec → 実装 → docstring化 → 仕様が安定 → スキル化
```

形式化が完了し、もはや変わらない不変条件だけがスキル化の対象になります。これは「仕様は実装に先んじる」が神話であることの帰結でもあります——スキル化できるほど安定した仕様は、実装を経て初めて生まれるからです。

## ブートストラップ問題

ここで「Spec部分はrustdoc/docstringに書くべきでは？」という疑問が生じます。その通りです。docstringはコードと同居するため腐りにくく、エージェントが参照する最も信頼できるソースです。

しかし冷静に考えると、コードを書くまでdocstringは存在しません。コードを書かせるためのドキュメント（実装用Spec）はどこに置くのか——これがブートストラップ問題です。

結局、3つの層が必要で、それぞれライフサイクルが異なります。

| 層 | 寿命 | 例 |
|---|---|---|
| 実装用Spec | 実装完了まで | 「Scalar traitをこう設計する」 |
| docstring | コードと共に生存 | `/// Trait for scalar types` |
| ADR/FS | 永続 | 「E0592回避のため補助トレイトを導入」 |

実装が完了したら実装用Specは**アーカイブ**するのが現実的です。docstringが正（canonical）になり、実装用Specはエージェントの通常の視界から外れる。消す必要はないが、デフォルトでは見えない場所に移す。

「実装後は仕様書を捨てろ」という[主張](https://zenn.dev/karamage/articles/a0b7111698ecb2)もあります。エージェントの視界から消すという意味では正しい方向ですが、物理的に削除すると遡及的な調査ができなくなります。アーカイブなら「消えているが、問い合わせれば出てくる」が成立します。

ただし、ファイルシステム上のディレクトリ移動（`archive/`に入れる）だけでは不十分です。`grep`や`rg`で検索すればアーカイブ済みファイルも同じ重みで引っかかります。デフォルト不可視を実現するには、後述するDB/CLI境界が必要になります。

なお、元記事もADRに対してステータス管理（Accepted/Superseded/Deprecated）を提案しています。アーカイブのトリガーとしてステータスを使い、`status: implemented`や`status: superseded`になったドキュメントを自動的にエージェントの視界から外す——という形で、ステータス管理とアーカイブを組み合わせるのが一つの方向です。

## エージェントのコンテキスト品質制御

ここまでの議論を踏まえると、記事が見落としている最も根本的な問題が見えてきます。

記事は「リンター」「Hooks」「テスト」という決定論的ツールでコードの品質を担保する話をしていますが、**エージェントが参照するドキュメントの品質を担保する仕組み**には触れていません。「CLAUDE.mdを小さく保て」は入口の制御に過ぎず、エージェントが自律的に検索した先の品質は無法地帯です。

生markdownファイル群を直接エージェントに検索させると、archiveのMLIR時代のドキュメントも現行のSpecも同じ重みで引っかかります。エージェントにとっては区別がつきません。これは記事が引用する「リポジトリ内で発見できる古い情報は最新の真実と区別できない」問題そのものです。

### DB境界というアプローチ

この問題に対する構造的な解は、エージェントとドキュメントの間にDB境界を設けることです。

- **検索の決定論化** — 同じクエリに対して常に信頼できるノードだけ返す
- **汚染の構造的排除** — 腐敗ノードはDBレベルで不可視にする。LLMの判断に委ねない
- **関係の明示化** — 「AはBに依存する」がスキーマで強制される

ドキュメント間の関係は自然にグラフ構造を取ります。

```
FS: E0592調査 ──発見──→ ADR: 補助トレイト採用 ──根拠──→ Spec: Scalar trait定義
                              │
                              ├── supersedes → ADR: Generic Float方式(却下)
                              └── depends_on → ADR: num-complex採用
```

RDBでもグラフDBでも、スキーマを導入した時点で「このデータは信頼できるか」「この関係は有効か」を問い合わせ可能になります。

### ドキュメンテーション第一原理との緊張

ただし、ここには緊張関係があります。エージェントに渡すべきドキュメントは可能なら公開ドキュメンテーション（rustdoc等）として整理すべき、という第一原理があります。

公開ドキュメントは「人に読まれる」という圧力があるからメンテされます。ADRにはその圧力がありません。だから腐ります。DB密封で守ろうとすると、公開ドキュメントを充実させれば不要になる知識まで永続化してしまいます。

本当に必要な優先順位は以下の通りです。

1. **rustdoc/docstringを充実させる** — エージェントが最も信頼できるソース。コードと同居するから腐りにくい
2. **ADR/FSは「公開ドキュメントでは伝えられないもの」だけ残す** — 却下した選択肢、非自明な制約の発見
3. **DB化はその残った最小限のADR/FSに対してのみ意味がある**

### Obsidian CLIという選択肢

frontmatter付きmarkdownの管理ツールとして、Obsidian CLIが有力な候補です。Obsidian 1.12以降、GUIなしでCLIから以下の操作が可能です。

- `obsidian backlinks` — 逆リンク（グラフのトラバーサル）
- `obsidian unresolved` — リンク切れ検出
- `obsidian property:read/set` — frontmatterのCRUD
- `obsidian orphans` — 被リンクなしファイル検出
- `obsidian tags` — タグベースのクエリ

dev-docsをObsidian vaultとして管理すれば、先述のグラフDB的な機能がそのまま使えます。MCPサーバを立てるまでもなく、CLIベースのスキルとしてエージェントに統合できます。

## まとめ

記事のハーネスエンジニアリングの枠組みは、コードの品質保証において優れた指針を提供しています。しかし、ドキュメント管理には以下の盲点があります。

1. **仕様書の居場所がない** — ADRは決定記録であり、仕様の定義は別の器が必要
2. **「仕様→実装」の直線モデルが暗黙の前提** — 実際はFSを通じて実装が仕様を発見する
3. **ドキュメントの分類が粗い** — FS・ADR・Specはそれぞれ腐敗特性が異なる
4. **エージェントのコンテキスト品質制御がない** — コードにリンターがあるように、ドキュメントにもアクセス制御層が必要

リンターがコードの品質を保証するように、DB境界がコンテキストの品質を保証する——これが記事の枠組みに対する補完的な視点です。
