## はじめに

Rustでテンソルネットワークライブラリを書き始めました。
現在はテスト込みで35 kloc程度まで育っていて、U(1)対称性つきBlockSparse DMRGが動くところまでは来ています。

つまり、少し触って「Rustむずかしい」と言っているわけではありません。
実装はそれなりに進みました。
進んだうえで、Rustとテンソルネットワーク計算の相性の悪さをかなり強く感じています。

一言でいうと、Juliaでは無料だと思っていた抽象が、Rustではすべて有料でした。

- `A[i, j]`と書くための料金
- `inner(a, b)`と書くための料金
- `@tensor C[i, j] := A[i, k] * B[k, j]`と書くための料金
- 実行時に形状が変わるテンソルを扱う料金
- 実数・複素数・BlockSparse・backendを一つの抽象に乗せる料金
- 分散並列で所有権を扱う料金

Julia村を出てRustへ上京して、初めてインフラのありがたさを思い知った、という感覚です。
この記事はその記録です。

## そもそもなぜ自前実装なのか

つらいなら既存のライブラリを使えばよいのでは、という話がまず出ます。
しかし、テンソルネットワークが必要とするものを軸に並べると、どれも素直には乗りません。
効くのは次の4つです。

- ランクが実行時に変わる(SVDやreshapeで軸の数そのものが変わる)
- 複素数(量子多体系では振幅が複素数)
- SVD / QR / 固有値分解(正準形とボンド次元の切り詰めの土台)
- U(1)対称性つきBlockSparse(量子数を保存するブロックだけを保持する、実用上の本体)

| ライブラリ | ランク | 複素数 | SVD/QR/固有値 | BlockSparse/対称性 |
|---|---|---|---|---|
| ndarray | 動的可(`IxDyn`) | ○(`num-complex`) | △(`ndarray-linalg`=LAPACK) | ✗ |
| Burn | 静的(次元数を型に持つ) | ✗ | ✗(LU止まり) | ✗ |
| candle | 動的 | ✗(予定なし) | ✗ | ✗ |
| tch | 動的 | ○ | ○ | ✗ |
| 欲しいもの | 動的 | ○ | ○ | ○ |

Burnは次元数(ランク)を型パラメータとして持ちます。
各軸のサイズは実行時の値ですが、ランクはコンパイル時に固定されます。
テンソルネットワークではSVDやreshapeでランクそのものが変わるので、これは効きます。

candleは形状もランクも実行時の値なので、Burnのこの問題は起きません。
「動的shapeがダメ」という理由ではcandleを外せない、というのが正直なところです。
ところが、candleには複素数型がなく、複素数サポートの要望にもメンテナは「予定なし」と明言しています(<https://github.com/huggingface/candle/issues/2965>)。
SVD・QR・固有値分解のネイティブ実装もなく、BlockSparseの概念もありません。
テンソルネットワークで一番難しい部分が丸ごと欠けています。

tchはPyTorch由来なので複素数もSVDも揃っています。
ただし、tchはlibtorch(PyTorchのC++コア)へのバインディングにすぎません。
そこへ全面的に乗るなら、Rustで書かずに素直にPythonからPyTorchを使えばよい、という話になります。
Rustでやる意味がほぼ消えるので、これは外します。
なお、U(1)対称性つきBlockSparseはtchにもないので、どのみち自前実装は残ります。

ndarrayは密配列の部品であり、線形代数はLAPACKバインディング、対称性は範囲外です。

結局、表のいちばん下の「欲しいもの」、とくにU(1)対称性つきBlockSparseは、どの汎用ライブラリも面倒を見てくれません。
そういうわけで自前実装になります。
以降は、それを書くと何にいくら請求されるか、という話です。

## 型関連メソッドを素直に定義しづらい

以前、Rustのinherent impl衝突について記事を書きました。

<https://zenn.dev/ultimatile/articles/rust-inherent-impl-conflict-workarounds>

テンソルライブラリでは、`f32`・`f64`と、それらの複素数型をまとめて扱いたくなります。
たとえば`Tensor<f64>`と`Tensor<Complex<f64>>`の両方に`norm()`や`svd()`のようなメソッドを生やしたい。

ところが、Rustでは型に直接定義するメソッド(inherent impl)が将来的に衝突しうる場合、E0592で拒否されます。
そのため、素直に「実数用の`impl`」「複素数用の`impl`」と分けることができません。
補助トレイトやニュータイプで回避する必要があります。

ここまでは、まあ受け入れられます。
Rustのコヒーレンスや将来互換性を守るための制約であり、作法として理解できます。
最初はコンパイラが何を言っているのか分かりませんでしたが、理解すれば納得はできます。

問題は、この回避策がテンソルライブラリ全体にじわじわ染み出すことです。

実数と複素数を一つの`Scalar`トレイトに束ねると、今度は実数と複素数の非対称性を自前で表現する必要があります。
たとえば`conj()`は実数では恒等写像ですが、複素数では共役を返します。
`im()`は複素数では虚部ですが、実数では常に0です。

数学的には自然な統一でも、型としてはかなり無理をしています。
`num-complex`や`num-traits`の外側に、自分のライブラリ用の数値タワーをもう一段作ることになります。

## 添字アクセスがきしょすぎる

テンソルを書いていると、当然こう書きたくなります。

```julia
A[i, j]
```

Rustではこれが書けません。
`A[i, j]`って有料だったんだ、という驚きがありました。

ここには別々の問題が2つあります。

まず、Rustの添字構文は添字式を1個だけ受け取ります。
そのため、`A[i, j]`という構文はありません。
多次元添字を渡すには、配列やタプルなどの1個の値にまとめる必要があります。

```rust
a[[i, j]]
a[(i, j)]
```

次に、Rustの添字アクセスは`Index`/`IndexMut`トレイトで定義されます。
`Index`は値ではなく参照を返します。

```rust
pub trait Index<Idx>
where
    Idx: ?Sized,
{
    type Output: ?Sized;

    fn index(&self, index: Idx) -> &Self::Output;
}
```

これは「既に存在する要素の場所を返す」モデルです。
CPUメモリ上の密な配列なら自然です。
しかし、GPUテンソル、遅延評価テンソル、BlockSparseテンソル、分散テンソルのように、添字アクセスが計算・転送・ブロック探索を伴う場合とは相性が悪い。
その場合は`get`/`set`を使うか、テンソルを返すスライス操作へ逃げます。

既存ライブラリの要素アクセスを眺めると、かなり思想が分かれます。

| ライブラリ | 言語 | 単一要素の読み取り | 単一要素の書き込み | 備考 |
|---|---|---|---|---|
| ndarray | Rust | `a[[i, j]]` / `a[(i, j)]` / `a.get([i, j])` | `a[[i, j]] = x` | `Index`/`IndexMut`が参照を返す。配列・タプル両方の添字が可 |
| nalgebra | Rust | `m[(i, j)]` / `m[k]` / `m.get((i, j))` | `m[(i, j)] = x` | タプル添字。参照を返す |
| candle | Rust | `t.i((i, j))?.to_scalar::<f32>()?` | なし | `.i(...)`はテンソルを返すので抽出ステップが要る |
| burn | Rust | `t.slice([i..i+1, j..j+1]).into_scalar()` | なし | スカラーは1要素テンソル経由 |
| tch | Rust | `t.double_value(&[i, j])` | `t.index_put_(...)` | `.get(i)`/`.i((i, j))`はテンソルを返す |
| NumPy | Python | `a[i, j]` | `a[i, j] = x` | `a[(i, j)]`の糖衣 |
| Julia | Julia | `A[i, j]` | `A[i, j] = x` | 1始まり・列優先 |
| PyTorch | Python | `t[i, j].item()` | `t[i, j] = x` | `[]`は0次元テンソルを返す |
| Eigen | C++ | `m(i, j)` | `m(i, j) = x` | `operator()`を使う。`[]`はベクトル専用で2添字不可 |

ここで効いているのは、言語仕様だけでなくストレージモデルの違いです。
`ndarray`や`nalgebra`、NumPy、Julia、Eigenは、要素を代入可能な「場所」として露出します。
一方、深層学習系のテンソルは、テンソルを不透明なバッファとして扱うため、要素アクセスがテンソル演算や抽出操作になります。

計算効率の観点からは、単一要素アクセスは可能なら避けるべきです。
したがって、candleやburnが単一要素の`[]`を積極的に提供しないのは、設計としては理解できます。

ただ、テンソルネットワークの実装では、デバッグや小さいテンソルの操作、BlockSparseのブロック確認などで、添字アクセスを書きたくなる場面が普通にあります。
そのたびに`A[i, j]`のありがたさを思い知ります。

## 二項演算がメソッドになるとオペランドが非対称になる

テンソルの内積を計算する`Inner`トレイトを定義したとします。
Rustで自然に書くと、呼び出しはこうなります。

```rust
a.inner(b)
```

しかし、数学的には本来こう書きたい。

```rust
inner(a, b)
```

メソッド呼び出しでは、`Self`が左辺に固定されます。
そのため、二項演算がAPI上は非対称になります。

もちろん自由函数を別途用意すれば`inner(a, b)`は作れます。
しかし、トレイトとして抽象化したいとき、ディスパッチの主体はどちらか、所有値と参照の組み合わせをどう扱うか、可換な演算でも左右を区別するのか、といった問題が出ます。

これは小さい問題に見えますが、テンソルライブラリではすぐ効いてきます。

- `Tensor`と`Tensor`
- `&Tensor`と`Tensor`
- `Tensor`と`&Tensor`
- `&Tensor`と`&Tensor`
- DenseとBlockSparse
- CPU backendとGPU backend

組み合わせが増えるほど、メソッド呼び出しの左辺に何を置くかがAPI設計上の圧力になります。
Juliaの多重ディスパッチでは自然に中央へ置けていたものを、Rustではどこかの型に所属させる必要があります。

## 動的サイズのテンソルが扱いづらい

既存ライブラリが乗らない理由のうち、動的shapeはそれ自体がRust固有のつらさにもなります。

テンソルネットワークでは、保持するテンソルの形状が計算中に適応的に変わります。
SVDでボンド次元を切り詰め、reshapeし、permuteし、縮約し、また切り詰める。
MPS/MPOの計算ではこれを延々と繰り返します。

型パラメータとしてランクを持つ設計は、コンパイル時にランクが分かっている場合には強力です。
しかし、テンソルネットワークでは実行時にランクも各軸のサイズも変わります。
結局、shapeの整合性は実行時に検査する必要があります。

つまり、Rustの型で守れる範囲を大きく超えます。
その結果、テンソル演算の各所で`Result`が増えます。

- ランクが合わない
- 添字ラベルが重複している
- 縮約する軸のサイズが合わない
- reshape後の要素数が合わない
- BlockSparseのsectorが保存則を満たさない

これらはコンパイル時には検出できず、実行時エラーになります。
Rustの型安全性が不要という話ではありません。
ただ、テンソルネットワークで本当に欲しい不変条件の多くは、普通のRust型では表現しきれません。

## 縮約一発が遠い

テンソルネットワークで最も基本的な操作は縮約です。
Juliaなら`TensorOperations.jl`でこう書けます。

```julia
@tensor C[i, j] := A[i, k] * B[k, j]
```

これは単に記法が短いだけではありません。
添字ラベルから縮約軸を読み取り、必要な並べ替えを行い、行列積に落とし、出力の添字順に戻す、という処理をまとめて表現しています。

Rustで同じことをやると、まず内部ではだいたい次のような段階を踏みます。

1. einsum風の式や添字ラベルを解析する
2. 入力テンソルの軸を縮約軸・非縮約軸に分ける
3. GEMMに落とせるように`permute`する
4. 2次元行列へ`reshape`する
5. BLASの`gemm`を呼ぶ
6. 結果を再び多次元テンソルへ戻す
7. 出力添字順に再度`permute`する

縮約は数学的には一行ですが、実装上は`permute + reshape + matmul + reshape + permute`です。
ここまではどの言語でも必要です。
つらいのは、Rustではその上に型・所有権・エラー処理・backend抽象が乗ることです。

さらにRustには可変長ジェネリクスがありません。
2入力の縮約は書けても、N個のテンソルを受け取る一般の縮約は別の設計が必要になります。
実用上は、再帰的に2入力縮約へ分解するか、動的な配列として受け取るか、マクロで押し込むかになります。

## 埋め込みDSLが書きづらい

TensorOperations.jlのような埋め込みDSLは、テンソルネットワークでは本当に便利です。

```julia
@tensor C[i, j] := A[i, k] * B[k, j]
```

Rustでも手続き的マクロを使えば、見た目だけは近いものを作れます。

```rust
tensor!(C[i, j] := A[i, k] * B[k, j]);
```

しかし、Juliaでは通常の式として構文解析されたASTをマクロで変換できます。
Rustでは、`macro!(...)`の中身として独自構文を受け取り、パースし、検査し、Rustコードを生成する必要があります。

さらに、よいDSLを作ろうとすると次が必要になります。

- 添字ラベルの出現回数チェック
- 縮約軸と出力軸の推論
- shape不一致のエラー表示
- borrow/ownershipを満たすコード生成
- backendごとの実装選択
- IDE支援やコンパイルエラーの見やすさ

これを書くくらいなら、普通にメソッドを書きます、という気持ちになります。

ここでも「実装できない」のではありません。
実装はできます。
しかし、個人開発のテンソルネットワークライブラリで、その実装コストを払う価値があるかというとかなり怪しい。

## ジェネリクスとFFIの境界で型情報が一度死ぬ

数値計算ライブラリは、最終的にBLAS/LAPACKやそれに類するC/Fortran由来のライブラリを呼びます。
ここでもRustのジェネリクスと外部ライブラリの境界がつらい。

Rust側では`T: Scalar`として、`f32`・`f64`・`Complex<f32>`・`Complex<f64>`をまとめて扱いたい。
しかし、BLAS/LAPACK側では函数が分かれています。

- `sgemm`
- `dgemm`
- `cgemm`
- `zgemm`

SVDや固有値分解でも、実数型と複素数型で入力・出力の型関係が変わります。
複素数行列の特異値は実数です。
Hermitian固有値問題の固有値も実数です。

Rustの型レベルでは`T::Real`や`T::Complex`のようなassociated typeで表現できます。
しかし、C ABIの前では最終的に具体型へ落とす必要があります。
そのため、実装は`TypeId`で分岐し、ポインタを対応する具体型へ再解釈して、適切なFFI函数を呼ぶ、という形になりがちです。

これはかなり嫌です。
Rust側で積み上げた型情報が、FFI境界で一度死にます。
そのあと、`unsafe`で「これは本当に`f64`のバッファです」と人間が宣言し直すことになります。

この層は、Rustの安全性と既存HPC資産の接続部分です。
そして、科学技術計算ではここを避けられません。

## BlockSparseの型パラメータが上位に漏れる

テンソルネットワークで本当に面白いところは、DenseテンソルではなくBlockSparseテンソルです。
U(1)対称性つきの計算では、各軸に量子数が載り、保存則を満たすブロックだけを保持します。

このとき、`BlockSparse<T, S>`のようにsector型`S`を型パラメータとして持たせると、`S`が上位の型へ伝播します。

たとえば、MPS、MPO、DMRGソルバ、演算子、backend抽象のすべてが`S: Sector`を背負い始めます。
Denseしか使わないコードまで、対称性セクターの型パラメータを意識することになります。

では`enum`でDenseとBlockSparseを包めばよいか。
それも簡単ではありません。
DenseテンソルとBlockSparseテンソルは、単なるストレージ違いではなく、数学的な意味が違います。
BlockSparseでは量子数付きindex、flux保存、bra/ket方向などが計算の整合性に関わります。
雑に`enum Tensor { Dense(...), BlockSparse(...) }`で包むと、型安全性を落としたうえで、各所に`match`ボイラープレートが増えます。

結局、trait-genericなテンソル抽象へ寄せることになります。
しかし、ここでもRustには高階型(HKT)がないので、backendとstorageとscalarの関係をきれいに表現するのは難しい。

テンソルネットワーク固有のつらさとしては、ここがかなり大きいです。
単に「Rustの構文が面倒」という話ではなく、数学的な構造を型に載せようとすると、型パラメータがライブラリ全体へ漏れていきます。

## backendをテンソルに持たせると所有権の菱形問題が起きる

GPUや分散を見据えると、テンソルは何らかのbackendを持つ必要があります。
素朴にはこうしたくなります。

```rust
struct Tensor<T, B> {
    data: Storage<T>,
    backend: Arc<B>,
}
```

しかし、MPSやMPOは多数のテンソルの集まりです。
各テンソルが`Arc<B>`を持つと、MPS全体のbackend権威はどこにあるのか、という問題が出ます。

MPSがbackendを持ち、各サイトテンソルもbackendを持つ。
さらに演算子や一時テンソルもbackendを持つ。
すると、所有権の菱形問題のようなものが起きます。

同じbackendを指しているつもりでも、API上は複数の権威が存在します。
別backend由来のテンソルを混ぜたとき、どこで検出するのか。
backendの寿命は誰が保証するのか。
一時テンソルを作る操作はどのbackendを使うのか。

この問題を避けるには、テンソル自体へbackendを持たせない設計に寄せます。
代わりに、call-siteから操作能力を渡します。
しかしそれはそれで、すべての操作にbackend/capabilityが露出します。

ここでも、Rustの所有権はローカルには強い一方で、数値計算ライブラリの「計算資源の権威」をどう表現するかは自分で設計する必要があります。

## 分散並列は所有権モデルから練り直しになる

ここまでは単一プロセスの話です。
分散並列を考えると、さらに一段つらくなります。

最初は「あとでMPI層を足せばよい」と思っていました。
しかし、テンソルを分散配置するなら、それは後付けでは済みません。
所有権モデルから練り直しになります。

ローカルテンソルでは、データの所有者はRustの値です。
スコープを抜ければdropされ、参照の生存期間はコンパイラが管理します。

分散テンソルではそうはいきません。

- データの実体は別ノードにあるかもしれない
- 複数ノードに複製されているかもしれない
- 通信中でまだ利用中かもしれない
- ランタイムが配置や移動を決めるかもしれない
- 障害時には所有者が消えるかもしれない

この時点で、通常の`&T`や`Arc<T>`では足りません。
Lamellarの`Darc`のような分散参照カウント相当の抽象から考える必要があります。

外部ランタイムに逃がせば解決するかというと、そうでもありません。
外部ランタイムは実行時に配置・複製・移動・再実行を決めたい。
Rustはコンパイル時に所有者・参照・破棄タイミングを確定したい。
所有権の決定主体がRustとランタイムに分裂します。

突き詰めると、Monarchのような分散実行ランタイム層を抱える必要が出てきます。
分散オブジェクトの寿命、配置、転送、非同期実行、障害処理、スケジューリング、可観測性。
これはもうテンソルネットワークライブラリではなく、分散実行基盤です。

個人開発の規模を超えています。

## CUDAとの和解も待っている

さらにGPU対応を考えると、CUDAとの和解調停が待っています。

RustのGPU周りは、シェーダーや`wgpu`方面はかなり整っています。
しかし、科学技術計算で欲しいのは「GPUで何かが動く」ことではありません。
欲しいのは、cuBLAS、cuSOLVER、cuSPARSE、cuTENSOR、NCCLのようなCUDA-X資産への安定した接続です。

ここには依存関係だけで複数の層があります。

- Rust側の`-sys` crateとsafe wrapper
- CUDA Toolkit
- NVIDIA driver
- PTX/SASSとGPU architecture
- cuBLAS/cuSOLVER/cuTENSOR/NCCLなどの個別ABI
- build scriptとリンク探索
- stream、event、allocator、workspace

しかもRustの所有権とCUDA streamの非同期実行も素直には噛み合いません。
カーネルをlaunchした時点では計算は終わっていません。
Rustのスコープを抜けたからといって、そのバッファを破棄してよいとは限りません。
eventやstreamまで含めた寿命管理が必要です。

JuliaではCUDA.jlやartifact/JLLがかなり隠してくれていた部分が、Rustでは急に見えます。
Rustで保証された所有権が、CUDA境界でまた手動管理に戻る感覚があります。

## それでも続けるのか

ここまで書くと、Rustでテンソルネットワークライブラリを書くのは失敗だったように見えます。
実際、研究開発の主戦場としてRustを選ぶには時期尚早だったかもしれない、という所感はあります。

ただし、すべてが無駄だったとは思っていません。
U(1)対称性つきBlockSparse DMRGが動くところまで来たので、少なくとも単一ノード・共有メモリ並列のテンソルネットワーク基盤としては成立しつつあります。

問題は、RustでJulia並みの数式表現を再現しようとしたり、分散ランタイムまで抱えようとしたりすると、個人開発の範囲を超えることです。

現実的には、スコープを切る必要があります。

- Rust版は単一ノードの高性能コアとして育てる
- APIは明示的なメソッドと自由函数で妥協する
- DSLは後回しにする
- 静的shapeによる完全な型安全性は狙わない
- 分散実行は未解決の設計課題として隔離する
- CUDA対応は依存関係と所有権モデルを別途慎重に設計する

Rustで実装できないわけではありません。
むしろ、実装はできます。
ただし、Juliaでは言語とエコシステムが持っていた抽象を、Rustではライブラリ作者が一つずつ買い直す必要があります。

`A[i, j]`は無料ではありませんでした。

そして、分散並列やCUDAまで含めると、請求額は個人開発者にとってかなり重いです。

## おわりに

Rustは悪い言語ではありません。
安全性、明示性、性能、エコシステムの安定感には大きな価値があります。

しかし、テンソルネットワーク計算では、欲しい抽象がかなり高水準です。
多次元添字、多重ディスパッチ、動的shape、数式DSL、BlockSparse、外部HPCライブラリ、GPU、分散所有権。
これらを全部ライブラリ側で抱えるのは重い。

Juliaで空気のように使っていた機能は、空気ではありませんでした。
Rustへ来て、そのインフラのありがたさを思い知りました。

このライブラリの開発は続けたいです。
ただ、Rustで科学技術計算、とくにテンソルネットワーク計算をやるなら、現状では「何をRustで抱え、何を諦めるか」をかなり早い段階で決める必要があります。

少なくとも、35 kloc書いてから気づくと結構つらいです。
