## はじめに

コーディングエージェントを用いた2026年4月の個人開発で使用しているワークフローの一例を紹介します。
他の方の類似の記事がたくさんありますが、一例としてご笑覧いただければと思います。
正直、所望の成果物が得られないことがままあり、めっちゃおすすめです！みたいな話ではないです。

主にGitHub issueを用いて開発を進めています。
これはSSOT問題と関連し、仕様・要件はドキュメントとして残したいが、コードベースには残したくない、という理由です。
GitHub issueなら明示的に読ませない限りエージェントのコンテキストに混入する可能性は低いです。
また`gh`コマンドを利用してコマンドラインから制御可能である点も便利です。

メインはClaude Code (Opus 4.6)を利用しており、レビュワーとしてCodexとCopilotを使用しています。
プランはClaudeがMax x5プランで、Proプランだと全然足りなくなっており厳しいです。
CodexはPlus, CopilotはProプランで利用しています。

まずやりたいことを思いついたらClaude Codeのプロンプトにやりたいことを伝えます。
この辺りの入り方はケースバイケースで、簡単な指示で表現できる場合もあれば、先に設計文書を書いてから指示する場合もあります。
大体「こういうことをやりたいが何かアイデアはあるか？」という問いかけです。
この時点でissue起票させることもあります。
アイデアが良さそうだと思ったら実装計画に入ります。
このときに使うのがresearchスキルです。

<https://github.com/ultimatile/research-and-implement-skill/blob/main/skills/research/SKILL.md>

researchと言うとちょっと大仰ですが、実装計画に対する仮説と検証プロセスを回してもらうスキルです。

ここで立てる仮説は以下のようなものです。

- 変更する必要のあるものは何か？
- どのような不変条件が保たれる必要があるか？
- 変更によって壊れる可能性がある場所はあるか？

このような箇所に意識的に調査してもらうことで、実装計画の精度を上げることができます。

スキルではこれをsubagentsを用いて並列で検証するように書いています。

これで実装計画が立てられたら、次にCodexでプランのレビューをしてもらいます。
